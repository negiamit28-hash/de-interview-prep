🎯 Architecture Design Scenarios

Generic / Any Company
Design a real-time analytics platform for an e-commerce company▶

Requirements: Real-time order tracking, fraud detection, personalized recommendations.

Solution:

  • Ingestion: Application events -> Event Hubs (partitioned by customer_id). CDC from order DB via Debezium -> Kafka -> Event Hubs.
  • Processing: Spark Structured Streaming on Databricks. Watermarks for late data. Stateful processing for session-level fraud scores.
  • Storage: Bronze (raw events) -> Silver (enriched, deduplicated) -> Gold (aggregated metrics) on Delta Lake / ADLS Gen2.
  • Serving: Gold tables -> Synapse serverless SQL for Power BI dashboards. Low-latency -> Cosmos DB for API-served recommendations.
  • Orchestration: Databricks Workflows for streaming jobs. ADF for batch backfill and data quality checks.
  • Governance: Unity Catalog for access control and lineage. Purview for broader data catalog.
Migrate an on-prem SQL Server DW to Azure▶
  1. Assess: Azure Migrate + Data Migration Assistant. Identify incompatible features, table sizes, dependencies.
  2. Choose target: Synapse Dedicated Pool (existing DW patterns) or Databricks Lakehouse (modernization).
  3. Data migration: ADF with Self-hosted IR for initial full load. Staged Copy via Blob for Synapse. Parallel copy with partitioning.
  4. Schema migration: Hash-distributed facts, replicated dimensions. Adapt stored procedures to Synapse T-SQL or Spark.
  5. Incremental sync: CDC or Tumbling Window triggers during cutover.
  6. ETL migration: SSIS -> Azure-SSIS IR (lift-and-shift) or ADF Mapping Data Flows (modernize).
  7. Validation: Row counts, checksums, business report reconciliation.
  8. Cutover: Dual-run period, then switch. DNS/connection string update.
Handle a data quality incident: duplicate records in production▶
  1. Assess impact: Which tables? How many? Since when? (Delta time travel to find the commit.)
  2. Contain: Pause downstream consumers. Mark Gold tables as unreliable.
  3. Root cause: Check idempotency. MERGE or blind INSERT? Retry re-processed data?
  4. Fix: Deduplicate with ROW_NUMBER() on business key, keep latest. MERGE to apply clean data.
  5. Prevent: Dedup in Silver layer. Primary key constraints. Great Expectations checks. Fix pipeline to use MERGE.
  6. Monitor: Alert on duplicate count threshold. Data quality tests in CI/CD.

PySpark + 20 Sources

"You ingest from 20+ sources. One sends malformed JSON. How do you prevent it from breaking the entire pipeline?"▶

Why they ask: Tests defensive programming and fault isolation.

  1. PERMISSIVE mode: spark.read.option('mode','PERMISSIVE').option('columnNameOfCorruptRecord','_corrupt_record')
  2. Quarantine pattern: Route _corrupt_record IS NOT NULL to a quarantine Delta table. Alert on count.
  3. Source isolation: Each source = separate pipeline stage. One failure doesn't cascade.
  4. Schema enforcement: Define StructType explicitly, don't rely on inference.
  5. Monitoring: Track quarantine counts by source per day. Spike = alert.
schema = StructType([StructField("id", StringType()), ...])
df = spark.read.schema(schema) \
    .option("mode", "PERMISSIVE") \
    .option("columnNameOfCorruptRecord", "_corrupt") \
    .json("abfss://raw@adls.dfs.core.windows.net/source1/")

good = df.filter(col("_corrupt").isNull()).drop("_corrupt")
bad = df.filter(col("_corrupt").isNotNull())
bad.write.mode("append").format("delta").save("/quarantine/source1/")
"Walk me through accumulators, broadcast variables, multi-level caching — when do you use each?"▶

Why they ask: Your resume lists these — they'll test depth.

TechniqueWhen to UseWhen NOT to Use
Broadcast variablesSmall lookups (<200MB) in joins/mapsLarge tables (OOM), frequently updated data
AccumulatorsCounting events/errors across tasksLogic control — can double-count on task retry
MEMORY_AND_DISKDataFrame used in 2+ downstream actionsDataFrames used only once
MEMORY_ONLYSmall, frequently accessed dataLarge DFs — eviction causes full recompute
DISK_ONLYVery large DFs that don't fit in memoryPerformance-critical hot paths
Pitfall

Accumulator values in transforms (map, filter) may double-count on retry. Only read after an action completes.

Event Hub + ADF

"You reduced ADF ingestion time by 50%. Walk me through exactly what you changed."▶

Why they ask: Verifying your resume metric is real. Have concrete numbers ready.

  1. DIU tuning: Override Auto to 32-64 DIUs for large files.
  2. Parallel copy: parallelCopies to match source partitions.
  3. Partition-aware copy: Dynamic range partitioning for large tables.
  4. Staged copy: Blob + PolyBase is 3-5x faster for Synapse.
  5. Pipeline parallelism: ForEach for independent sources, not sequential.
  6. TTL on IR: Eliminate Managed VNet cold start between activities.
  7. Compression: Ingest gzipped files — smaller transfer.
Interview Tip

"We had 20 sources running sequentially. I parallelized them in ForEach, increased DIUs from auto (4) to 32, and enabled staged copy for Synapse. Wall-clock time went from 4 hours to under 2." Specific numbers = credible.

"Event Hub has at-least-once delivery. How did you handle duplicates in ADLS?"▶
  • Consumer: Enable checkpointing (offset tracking in Blob).
  • Bronze: Accept duplicates (append-only, raw fidelity).
  • Silver: Delta MERGE on composite key (business_key + event_id).
  • Streaming: dropDuplicatesWithinWatermark("event_id") for bounded-state dedup.
MERGE INTO silver.events t
USING bronze_batch s
ON t.event_id = s.event_id AND t.business_key = s.business_key
WHEN NOT MATCHED THEN INSERT *
"Your PySpark job uses multi-level caching. After adding a 4th join, executors OOM. What happened?"▶

Root cause: Cached DataFrames + 4th join's shuffle data exceeded executor memory. Spark's unified memory splits between storage (cache) and execution (shuffle). Over-caching squeezes execution.

Fix:

  1. df.unpersist() after last use of each cached DataFrame.
  2. Cache only DataFrames reused in 2+ actions.
  3. Switch large caches to DISK_ONLY.
  4. Tune spark.memory.storageFraction (default 0.5) — lower it to give execution more room.
  5. Check if the 4th join introduces skew amplifying memory pressure.
"You implemented Dimensional Modelling for OLAP. A customer moved states — how does your Star Schema handle historical queries?"▶

SCD Type 2: Two rows in customer dimension with effective dates.

-- Customer moved from NY to NJ on 2025-07-01
dim_customer:
  customer_sk=101, customer_id='C001', state='NY',
    effective_date='2020-01-01', end_date='2025-06-30', is_current=0

  customer_sk=102, customer_id='C001', state='NJ',
    effective_date='2025-07-01', end_date='9999-12-31', is_current=1

-- Last year's claims join to sk=101 (NY)
-- Current claims join to sk=102 (NJ)
-- Fact table uses surrogate key, not natural key

Requires surrogate keys since natural key (customer_id) now has multiple rows. Fact table's FK points to the correct version based on claim date.

💼 Accenture Scenarios (2019-2022)

Resume-Based
"You built Kafka streaming pipelines. How did you ensure exactly-once semantics end-to-end?"▶

Three components required:

  1. Source (Kafka): Replayable — consumer tracks offsets.
  2. Engine (Spark Structured Streaming): Checkpoint directory tracks committed offsets + state per micro-batch.
  3. Sink (Delta Lake): Idempotent — transaction log prevents duplicate writes.
df = spark.readStream.format("kafka") \
    .option("kafka.bootstrap.servers", "broker:9092") \
    .option("subscribe", "claims-topic") \
    .load()

df.writeStream.format("delta") \
    .option("checkpointLocation", "/checkpoints/claims/") \
    .outputMode("append") \
    .trigger(processingTime="30 seconds") \
    .start("/silver/claims/")
Key Point

"Exactly-once" = "effectively-once". Messages may be delivered multiple times, but the effect is applied once.

"Your Kafka consumer lag is increasing. Systematic troubleshooting?"▶
  1. Check lag per partition — is it one or all?
  2. Profile processing: CPU-bound? I/O-bound? Sink bottleneck?
  3. Optimize transforms (reduce serialization, efficient formats).
  4. Scale consumers up to partition count (max parallelism = partition count).
  5. Batch sink writes, use async I/O for ADLS.
  6. If partitions maxed, increase count (triggers rebalance).
  7. Implement backpressure: maxOffsetsPerTrigger in Spark.
"Autoloader with 1000s of small files — performance degrades. Fix?"▶
  • Switch from directory listing to file notification mode (Event Grid) — instant detection.
  • Set maxFilesPerTrigger to control batch size.
  • Schedule OPTIMIZE on Delta tables to compact small files.
  • Enable autoCompact and optimizedWrite table properties.
# Autoloader with file notification
df = spark.readStream.format("cloudFiles") \
    .option("cloudFiles.format", "json") \
    .option("cloudFiles.useNotifications", "true") \
    .option("cloudFiles.maxFilesPerTrigger", 1000) \
    .load("abfss://raw@adls.dfs.core.windows.net/source/")
"You automated Databricks jobs via REST APIs. Walk through the automation and failure handling."▶
  1. POST /api/2.1/jobs/create — define notebook, cluster, schedule.
  2. POST /api/2.1/jobs/run-now — trigger with parameter overrides.
  3. GET /api/2.1/jobs/runs/get?run_id=xxx — poll until TERMINATED.
  4. On FAILED: capture state.state_message, auto-rerun with /runs/repair, alert.
  5. POST /api/2.1/jobs/runs/delete — cleanup old runs.

💼 Cognizant Scenarios (2016-2018)

Resume-Based
"Walk me through the Netezza-to-Azure migration. What were the biggest challenges?"▶

Approach:

  1. Assessment: Inventory tables, row counts, data types. Map Netezza types (NCHAR, BYTEINT) to Azure equivalents.
  2. Schema: Netezza distribution keys -> Synapse hash distribution. Zone maps -> columnstore indexes.
  3. Data: ADF + Self-hosted IR. Partition large tables for parallel copy. Blob staging -> PolyBase load.
  4. Logic: NZPLSQL -> T-SQL or PySpark. U-SQL for intermediate transforms (deprecated now).
  5. Validation: Row counts, checksums, sample data comparison, business report validation.

Challenges:

  • Data type mismatches (NUMERIC precision differs)
  • Performance regression (zone map queries need different indexing)
  • Character encoding (Latin-1 vs UTF-8)
  • Network bandwidth for multi-TB migrations

💼 Infosys Scenarios (2018-2019)

Resume-Based
"You designed Star Schema models. Walk me through a real example from insurance/financial domain."▶ 
-- Fact: Insurance Claims
CREATE TABLE fact_claims (
    claim_sk       BIGINT,          -- surrogate key
    date_sk        INT,             -- FK to dim_date
    customer_sk    INT,             -- FK to dim_customer (SCD Type 2)
    policy_sk      INT,             -- FK to dim_policy
    provider_sk    INT,             -- FK to dim_provider
    claim_amount   DECIMAL(18,2),   -- measure
    paid_amount    DECIMAL(18,2),   -- measure
    claim_count    INT DEFAULT 1,   -- for aggregation
    claim_number   VARCHAR(20)      -- degenerate dimension
);

-- Dimension: Customer (SCD Type 2)
CREATE TABLE dim_customer (
    customer_sk    INT IDENTITY,
    customer_id    VARCHAR(20),     -- business key
    name           VARCHAR(100),
    state          VARCHAR(2),
    effective_date DATE,
    end_date       DATE,            -- 9999-12-31 for current
    is_current     BIT
);

Key design decisions:

  • Grain: One row per claim (transaction fact). Declare grain first.
  • SCD Type 2 on customer: Address/plan/status changes matter for analytics.
  • Degenerate dimension: claim_number in fact (no separate dim).
  • Conformed dimensions: dim_date, dim_customer shared across fact tables.

⚠️ Resume Gap Questions

Prepare for These

Interviewers will notice what's not on your resume and probe these areas.

"Your resume doesn't mention Delta Lake, Unity Catalog, or CI/CD for ADF. Are you familiar?"▶

Strategy: Be honest, show awareness, pivot to related experience.

  • Delta Lake: "We used Parquet on ADLS. I understand Delta's value — ACID, OPTIMIZE for compaction, MERGE for upserts. I've worked with these in development environments."
  • Unity Catalog: "We used workspace-level ACLs. I know UC provides catalog.schema.table namespace, centralized GRANT/REVOKE, and automatic lineage. I understand the migration path."
  • ADF CI/CD: "We managed multi-environment deployments. I understand Git -> ARM templates -> CI/CD with parameterized linked services. I used PowerShell deployment scripts at Cognizant."
Golden Rule

Never bluff. "I haven't implemented X in production, but I understand the concepts and have worked with Y which is related" is always better than fake depth.

"ADLA and U-SQL are retired. How would you modernize those workloads?"▶
  • U-SQL queries -> Spark SQL / PySpark on Databricks. C# extensibility -> PySpark UDFs.
  • ADLA compute -> Databricks serverless or Synapse Serverless SQL (OPENROWSET).
  • ADLA jobs -> Databricks Workflows or ADF + Databricks notebook activities.
  • PowerShell deployment -> Databricks Asset Bundles or Terraform.

"My ADLA experience gave me strong serverless query-over-files fundamentals. The modern equivalent is Databricks or Synapse Serverless — same concept, better performance."

"You have AZ-900 and DP-900 certifications. Have you considered DP-203 (Data Engineering Associate)?"▶

Good answer: "Yes, DP-203 is on my roadmap. My hands-on experience with ADF, Databricks, ADLS, and Event Hub covers the core exam topics. I plan to certify to validate that experience formally."

DP-203 Topics

Design and implement data storage (25-30%), data processing (25-30%), data security and governance (20-25%), monitor and optimize (15-20%). Heavy on Synapse, Databricks, ADF, ADLS — all in your experience.

"Compliance audit: who accessed PII data in the last 90 days? Prove source-to-report lineage."▶

Multi-layer approach (common for insurance companies):

  1. ADLS diagnostic logs -> Log Analytics for who-read-what-when.
  2. Unity Catalog lineage for table-to-table data flow tracking.
  3. ADF Monitor for pipeline-level data flow.
  4. Databricks audit logs for notebook/query access.
  5. PII protection: Column-level ACLs, dynamic views, sensitive column tagging in catalog.

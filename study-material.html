⚙ Azure Data Factory

Architecture & Core Concepts

ADF is a cloud-based, serverless PaaS for ETL/ELT data integration. It orchestrates data movement and transformation across hybrid environments with 100+ connectors.

Core Entities

  • Pipelines — Logical groupings of activities (max 80 per pipeline). Can be nested via Execute Pipeline activity.
  • Activities — Three categories: Data movement (Copy), Data transformation (Data Flow, Databricks, SP), Control flow (ForEach, If, Switch, Until, Lookup, etc.)
  • Datasets — Schema representations pointing to data. Parameterizable. Eliminated in Fabric (inline properties).
  • Linked Services — Connection definitions with auth. Support parameterization for multi-environment configs.
  • Triggers — Schedule, Tumbling Window (retry + dependency support), Event-based (blob creation), Custom Event (Event Grid topics).

Integration Runtimes (IR)

Senior-Level Insight

Azure IR auto-resolve picks the closest region to the sink for Copy, but uses the factory region for Data Flows. For data compliance (GDPR), pin IR to a specific region.

Parameterization & Dynamic Content

  • Pipeline parameters — passed between parent/child pipelines via Execute Pipeline
  • Dataset parameters — single dataset serves multiple tables: @dataset().tableName
  • Linked Service parameters — dynamically change connection targets per environment
  • Global parameters — Factory-level constants, overridable per environment in ARM parameter files
  • System variables — @pipeline().RunId, @pipeline().TriggerTime, @trigger().outputs.windowStartTime
  • Expression language — 200+ functions: @concat(), @formatDateTime(), @activity('name').output
Best Practice

Combine parameterized linked services + Key Vault references + global parameters for environment-agnostic pipelines. A single ADF codebase targets dev/test/prod by only changing the Key Vault name parameter.

CI/CD with ADF

  • Only dev connects to Git (Azure DevOps or GitHub). Test/prod deploy via CI/CD.
  • adf_publish branch: Auto-generated ARM templates on publish. Alternative: @microsoft/azure-data-factory-utilities npm package for CI-triggered generation.
  • Deployment flow: Dev → ARM template → Pre-deployment script (stop triggers) → Incremental ARM deploy → Post-deployment script (restart triggers) → Repeat for prod.
  • Never use Complete mode for ARM deployment (deletes resources not in template).
  • IR names must be identical across all stages.
  • Cherry-picking/selective publishing is not supported by design.

Performance Tuning

Data Integration Units (DIUs)

  • Measure CPU + memory + network for Azure IR. Range: 4-256.
  • Auto setting pitfall: optimizes by file count, not file size. Large files may be underprovisioned — override manually.
  • Billing: DIUs used × duration × unit price/DIU-hour.

Parallel Copy

  • parallelCopies controls max threads within a single Copy activity.
  • Orthogonal to DIUs — parallelism operates within DIU allocation.
  • Self-hosted IR: max effective parallel copy is 4-5 per node for Blob/ADLS Gen2.

Staged Copy

  • Uses Blob/ADLS Gen2 as interim staging for PolyBase/COPY into Synapse.
  • Enables compression during on-prem transfers and workaround for port restrictions (443 only).

TTL for Managed VNet

  • Keeps compute warm between activities (cold start takes minutes).
  • Critical for ForEach loops. Billing continues during TTL window.

Monitoring & Troubleshooting

  • ADF retains pipeline run data for 45 days only — route to Log Analytics for long-term retention.
  • Use KQL queries for advanced analysis of durations, failures, and resource consumption.
  • Copy activity monitoring: DIUs used, throughput (MB/s), duration breakdown (queue vs transfer).
  • Set up Azure Monitor metric alerts on PipelineFailedRuns, ActivityFailedRuns.

⚡ Databricks & PySpark

Databricks Architecture

  • Workspace: Collaboration environment with notebooks, repos, jobs, SQL warehouse.
  • All-Purpose Clusters: Persistent, interactive development and shared workloads.
  • Job Clusters: Auto-created per job run, terminated after — cost-efficient for production.
  • Instance Pools: Pre-warmed VMs to reduce cluster startup time.
  • Unity Catalog: Centralized governance — 3-level namespace (catalog.schema.table), access control, lineage.

Delta Lake Deep Dive

Open-source storage layer providing ACID transactions on top of Parquet files in data lakes.

Key Features

FeatureDescription
ACID TransactionsSerializable isolation via transaction log (_delta_log). Concurrent reads and writes.
Time TravelSELECT * FROM t VERSION AS OF 5 or TIMESTAMP AS OF '2026-01-01'. Audit, rollback, reproducibility.
Schema EnforcementRejects writes that don't match table schema. mergeSchema option for evolution.
OPTIMIZECompacts small files into larger ones (~1GB target). Run regularly for read performance.
VACUUMRemoves files no longer referenced. Respects retention period (default 7 days). Run after OPTIMIZE.
Z-OrderingCo-locates related data in files for predicate pushdown: OPTIMIZE t ZORDER BY (col)
Liquid ClusteringIncremental clustering without full rewrites. Adapts to query patterns. Replaces Z-ordering for new tables.

PySpark Performance

Catalyst Optimizer

  1. Parse → unresolved logical plan
  2. Analyze → resolve column/table references
  3. Rule-based optimization → predicate pushdown, column pruning, constant folding
  4. Cost-based optimization → generate physical plans, pick cheapest

This is why DataFrame API and SQL have identical performance.

Adaptive Query Execution (AQE)

  • Coalesce partitions: Merges small post-shuffle partitions to reduce task overhead.
  • Switch join strategy: Sort-merge → broadcast hash join if one side is small at runtime.
  • Handle skew: Splits skewed partitions for parallel processing.
  • Enabled by default since Spark 3.2. Collects runtime statistics during shuffle stages.

Narrow vs Wide Transformations

Narrow (No Shuffle)Wide (Shuffle Required)
map, filter, union, selectgroupBy, join, repartition, distinct, orderBy
Each partition → one outputPartitions → multiple outputs (network transfer)

Data Skew Handling

  1. AQE skew join: Automatic in Spark 3.2+ with AQE enabled.
  2. Key salting: Add random prefix to hot keys, join on salted key, then aggregate to remove salt.
  3. Broadcast join: If one side is small, df.join(broadcast(small_df), 'key').
  4. Isolate hot keys: Process hot keys separately, union results.

Partitioning Best Practices

  • Target 128MB-256MB per partition for optimal task size.
  • Use repartition(n) for even distribution, coalesce(n) to reduce without shuffle.
  • Partition by high-cardinality columns that are commonly filtered (e.g., date).
  • Avoid over-partitioning (too many small files) or under-partitioning (OOM).

Structured Streaming

  • Triggers: processingTime ('10 seconds'), once (single micro-batch), availableNow (process all available, then stop), continuous (experimental low-latency).
  • Watermarks: withWatermark('timestamp', '10 minutes') — defines max expected lateness for window aggregations.
  • Checkpointing: Writes progress to durable storage (ADLS/S3). Enables exactly-once with idempotent sinks.
  • Output modes: append (new rows only), complete (full result), update (changed rows).

Unity Catalog

  • 3-level namespace: catalog.schema.table (replaces hive_metastore)
  • Access control: GRANT/REVOKE at catalog, schema, table, column level
  • Lineage: Automatic tracking of data flow between tables
  • Audit logs: Who accessed what, when
  • Cross-workspace: Single governance layer across all workspaces in account

Photon Engine

C++ native vectorized execution engine replacing parts of JVM-based Spark. Accelerates SQL queries, joins, and aggregations on Delta tables. Available on Photon-enabled cluster types. No code changes needed — it's transparent.

Common Interview Questions

View Databricks/PySpark Interview Questions▶
  1. Explain Delta Lake ACID transactions. — Transaction log (_delta_log) with JSON entries for each commit. Optimistic concurrency control. Readers see consistent snapshot.
  2. When to use Liquid Clustering vs Z-ordering? — Liquid Clustering for new tables (incremental, adapts). Z-ordering for existing tables (requires full OPTIMIZE, static columns).
  3. How does Spark handle memory? — Unified memory: execution (shuffle, joins) + storage (cached data) share a pool. Execution can evict storage. Tune spark.memory.fraction.
  4. Explain the difference between repartition and coalesce. — Repartition: full shuffle, can increase or decrease partitions. Coalesce: no shuffle, can only decrease, may result in uneven partitions.
  5. Design a streaming pipeline for real-time fraud detection. — Event Hubs → Structured Streaming with watermarks → stateful processing (mapGroupsWithState) → Delta table → alerts.
  6. How do you handle small file problem in Delta Lake? — OPTIMIZE for compaction, auto-optimize (optimizedWrite + autoCompact), Liquid Clustering for ongoing compaction.

🗂 Advanced SQL

Window Functions

-- Ranking
SELECT name, score,
  ROW_NUMBER() OVER (ORDER BY score DESC) as rn,     -- 1,2,3,4
  RANK()       OVER (ORDER BY score DESC) as rnk,    -- 1,2,2,4 (gap)
  DENSE_RANK() OVER (ORDER BY score DESC) as drnk,   -- 1,2,2,3 (no gap)
  NTILE(4)     OVER (ORDER BY score DESC) as quartile -- Divide into 4 groups
FROM students;

-- Analytics
SELECT date, revenue,
  SUM(revenue) OVER (ORDER BY date ROWS UNBOUNDED PRECEDING) as running_total,
  AVG(revenue) OVER (ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) as moving_avg_7d,
  LAG(revenue, 1)  OVER (ORDER BY date) as prev_day,
  LEAD(revenue, 1) OVER (ORDER BY date) as next_day,
  revenue - LAG(revenue,1) OVER (ORDER BY date) as day_over_day_change
FROM daily_sales;

CTEs & Recursive CTEs

-- Recursive CTE: Org chart hierarchy
WITH RECURSIVE org_tree AS (
  -- Anchor: top-level manager
  SELECT id, name, manager_id, 1 as level
  FROM employees WHERE manager_id IS NULL
  UNION ALL
  -- Recursive: each subordinate
  SELECT e.id, e.name, e.manager_id, t.level + 1
  FROM employees e
  JOIN org_tree t ON e.manager_id = t.id
)
SELECT * FROM org_tree ORDER BY level, name;

MERGE Statement

MERGE INTO target t
USING source s ON t.id = s.id
WHEN MATCHED AND s.updated > t.updated THEN
  UPDATE SET t.name = s.name, t.amount = s.amount
WHEN NOT MATCHED BY TARGET THEN
  INSERT (id, name, amount) VALUES (s.id, s.name, s.amount)
WHEN NOT MATCHED BY SOURCE THEN
  DELETE;
Pitfall

If source has duplicate join keys, MERGE behavior is non-deterministic. Always deduplicate the source CTE before the MERGE.

Synapse Distribution Strategies

StrategyBest ForDetails
HashLarge fact tablesDistribute on join key. Co-locates data, minimizes movement. Choose high-cardinality column.
Round-robinStaging / temp tablesEven spread across distributions. Default. Data movement for every join.

Index Types

  • B-tree (Clustered/Nonclustered): Standard for OLTP point lookups and range scans.
  • Columnstore: Column-oriented, 10x compression. Ideal for analytics (large scans, aggregations). Use clustered columnstore for DW fact tables.
  • Covering Index: Includes all columns needed by a query → avoids key lookup. CREATE INDEX ix ON t(a,b) INCLUDE(c,d)
  • Materialized Views: Pre-computed aggregations. Synapse auto-maintains and auto-selects them in query plans.

Query Optimization Tips

  • Read execution plans — look for scans (bad), seeks (good), sort warnings, hash spills.
  • Use SET STATISTICS IO ON to see logical reads.
  • Avoid SELECT * — column pruning is critical for columnstore.
  • SARGable predicates: WHERE col > 5 (good) vs WHERE YEAR(col) = 2026 (bad, wraps column in function).
  • Update statistics regularly: UPDATE STATISTICS table.

📊 Data Modeling

Dimensional Modeling (Kimball)

Star Schema

  • Fact tables: Contain measures (revenue, quantity) and foreign keys to dimensions. Types: transactional, periodic snapshot, accumulating snapshot.
  • Dimension tables: Descriptive attributes (customer name, product category). Denormalized for query simplicity.
  • Central fact surrounded by dimensions — "star" shape. Preferred for modern cloud DWs.

Snowflake Schema

  • Dimensions are normalized into sub-dimensions (e.g., product → category → department).
  • Less storage redundancy but more complex joins. Rarely preferred in modern practice.

Slowly Changing Dimensions (SCD)

Type 0Never changeOriginal onlyDate of birth, SSN
Type 1OverwriteNoneCorrecting errors, unimportant changes
Type 2New row + effective dates + is_current flagFullCustomer address changes, most common
Type 3Add column for previous valueOne priorWhen only previous value matters

SCD Type 2 Implementation

Requires surrogate keys (natural key now has multiple rows). Add columns: effective_date, end_date (9999-12-31 for current), is_current flag. Use MERGE to detect changes and manage row versioning.

Data Vault 2.0

ComponentPurposeContent
HubBusiness entity identityBusiness key, hash key, load date, source
LinkRelationships between hubsHash key, FK to hubs, load date, source
SatelliteDescriptive attributes + historyFK to hub/link, attributes, load date, source, hash diff

Advantages: Insert-only (no updates), parallel loading, full audit trail, handles source changes gracefully. Best for: many source systems, regulatory audit requirements, agile development.

Medallion Architecture (Bronze / Silver / Gold)

  • Bronze: Raw ingestion. Append-only, schema-on-read. Preserves source fidelity. Partitioned by ingestion date.
  • Silver: Cleaned, deduplicated, standardized, schema-enforced. Conformed dimensions. Business rules applied.
  • Gold: Business-ready. Star schemas, aggregations, feature tables, KPI datasets. Optimized for BI tools.

De facto standard in Databricks/lakehouse. Each layer adds data quality. Supports data lineage and governance.

Advanced Concepts

  • Surrogate vs Natural Keys: Surrogate keys (auto-generated integers) are preferred for DWs — stable, compact joins, enable SCD Type 2. Natural keys (business keys) for operational systems.
  • Degenerate Dimensions: Transaction identifiers (invoice #, order #) stored directly in fact table without a dimension table.
  • Junk Dimensions: Combine low-cardinality flags/indicators (yes/no, status codes) into a single dimension table to avoid many small dimensions.
  • Bridge Tables: Handle many-to-many relationships between fact and dimension (e.g., multiple patients per encounter). Contains weighting factors.
  • Late-Arriving Facts: Facts arrive before their dimension context. Use default/unknown dimension members, then update when dimension data arrives.
  • Late-Arriving Dimensions: Dimension changes arrive after facts referencing them. Restate the dimension (SCD Type 2 with backdated effective dates) and reprocess affected facts.

📁 Azure Data Lake Storage Gen2

Architecture

  • ADLS Gen2 = Azure Blob Storage + Hierarchical Namespace (HNS).
  • HNS enables atomic directory operations (rename/delete in O(1)), POSIX ACLs, and better analytics performance.
  • Same pricing as Blob Storage. Same APIs (Blob REST, ABFS driver for Hadoop).
  • ABFS driver: abfss://container@account.dfs.core.windows.net/path

Storage Tiers & Lifecycle Management

ColdHigherLower still90 daysRarely accessed
ArchiveHighestLowest180 daysCompliance, long-term backup (hours to rehydrate)

Lifecycle management policies: Automatically tier data based on last modified/accessed date. E.g., move to Cool after 30 days, Archive after 90 days.

Security Model

RBAC vs ACLs

  • RBAC: Azure role-based access at container/account level. Evaluated first — if granted, ACLs are skipped.
  • ACLs: POSIX-style, path-level (specific folders/files). Access ACLs + Default ACLs (inherited by new children).
  • Best practice: RBAC for service principals and broad access. ACLs for fine-grained, user-level folder permissions.

Encryption & Network Security

  • Encryption at rest: Microsoft-managed keys (default) or customer-managed keys (CMK) via Key Vault.
  • Encryption in transit: TLS 1.2+ enforced.
  • Private endpoints: Access via Azure Private Link (no public internet).
  • Firewall: IP rules, VNet service endpoints, trusted Azure services.

Performance Optimization

  • File size: Target 256MB-1GB. Avoid small files problem (<1MB each).
  • Folder structure: Partition by date: /data/year=2026/month=04/day=06/. Avoid too-deep nesting.
  • Parquet format: Columnar, compressed, predicate pushdown. Row group size ~128MB.
  • Delta Lake on ADLS: Adds ACID, time travel, compaction (OPTIMIZE). De facto standard for lakehouse.

Azure Event Hubs

  • Architecture: Partitioned event log (like Kafka topics). Each partition is an ordered sequence.
  • Partitions: Unit of parallelism. 1-32 per Event Hub (Basic/Standard), up to 2000 (Premium/Dedicated).
  • Consumer Groups: Independent readers. Each group maintains its own offset. Max 5 (Standard) or 100 (Premium).
  • Capture: Auto-archive to ADLS/Blob in Avro format. Zero-code data lake ingestion.
  • Throughput Units (TU): 1 TU = 1 MB/s ingress, 2 MB/s egress. Auto-inflate for scaling.
  • Kafka compatibility: Event Hubs Premium/Dedicated supports Kafka protocol natively.

Spark Structured Streaming

  • Micro-batch model: Processes streaming data as a series of small batch jobs on DataFrames.
  • Watermarks: withWatermark('event_time', '10 minutes') — max expected lateness. State beyond watermark is dropped.
  • Checkpointing: Write-ahead log to ADLS/S3. Enables exactly-once with idempotent sinks.
  • Triggers: processingTime('10 seconds'), once=True (single batch), availableNow=True (process all, then stop).
  • Output modes: append (new rows), complete (full result), update (changed rows).
  • Stateful processing: mapGroupsWithState and flatMapGroupsWithState for custom session windows, complex event patterns.

Change Data Capture (CDC)

  • What: Captures row-level INSERT/UPDATE/DELETE from source database transaction logs.
  • Why: Low source impact (reads logs, not tables), real-time replication, event-driven architectures.
  • Tools: Debezium (open source, Kafka Connect), Azure SQL CDC, Oracle GoldenGate, AWS DMS.
  • Pattern: Source DB → CDC → Event Hub/Kafka → Spark Structured Streaming → Delta Lake (with MERGE for upserts).

Exactly-Once vs At-Least-Once

🐍 Python for Data Engineering

Advanced Python Concepts

Generators

  • Use yield for lazy evaluation — critical for large datasets. pd.read_csv(chunksize=N) returns a generator.
  • Chain generators: read_rows() → filter_valid() → transform() — each a generator, memory stays flat.
  • yield from: delegates to sub-generators. send() and throw() for coroutine-style communication.

Decorators

  • Three-layer pattern: decorator with arguments = outer factory → decorator → wrapper.
  • Always use functools.wraps to preserve function metadata.
  • Data engineering uses: retry logic, timing/logging, connection management, schema validation.

Context Managers

  • __enter__ / __exit__ protocol. with guarantees cleanup even on exceptions.
  • contextlib.contextmanager: convert generator with single yield into context manager.
  • contextlib.ExitStack: manage dynamic number of resources.

Descriptors

  • __get__, __set__, __delete__ protocol. Powers property, classmethod, staticmethod.
  • Data descriptors (with __set__) take precedence over instance __dict__.
  • Use for reusable field validators in data models.

Testing for Data Pipelines

  • pytest fixtures: function/module/session scope. yield fixtures for setup/teardown. conftest.py for shared fixtures.
  • moto: Mock AWS services (S3, DynamoDB) — invaluable for data engineering tests.
  • responses: Mock HTTP requests.
  • freezegun: Mock datetime for time-dependent pipeline tests.
  • pandera: DataFrame schema validation — define expected dtypes, value ranges, custom checks.
  • great_expectations: Declarative data quality assertions.

Design Patterns for Pipelines

  • Factory: Create extractors/loaders based on source type/config.
  • Strategy: Interchangeable transform logic (full load vs incremental, different validations).
  • Template Method: Base Pipeline class with abstract extract(), transform(), load().
  • Metadata-driven: Schema/transformation rules in YAML/JSON config, not hard-coded.
  • Idempotent loads: Every run produces same result regardless of repetition. Use MERGE/upsert.

Production Best Practices

  • Structured logging: JSON format with structlog. Include correlation IDs (run_id, task_id).
  • Error handling: Catch specific exceptions. Dead letter queues for failed records. Circuit breaker (pybreaker) for external services.
  • Dependency management: uv (Rust-based, fastest) for new projects; poetry for existing. Always use lockfiles.
  • Type hints: Protocol for interfaces, TypedDict for config shapes. mypy/pyright for static checking.
  • Performance: Profile first (cProfile, Scalene). Vectorize Pandas ops. lru_cache for expensive pure functions. numba.jit for numerical loops.

Common Interview Questions

View Python Interview Questions▶
  1. Design a pipeline ingesting from 5 REST APIs with different rate limits. — asyncio with per-API Semaphores, tenacity retries, dead letter queue for failures, idempotent loads.
  2. Process a 100GB file on 8GB RAM machine. — Generator-based chunked reading, Polars streaming mode, or PyArrow dataset with predicate pushdown.
  3. Explain the GIL. When does it matter? — Matters for CPU-bound threads (use multiprocessing). Doesn't matter for I/O-bound (threading) or single-threaded async.
  4. Compare Pandas vs Polars vs DuckDB. — Pandas: ecosystem. Polars: speed + lazy eval. DuckDB: SQL interface + columnar, zero-copy with Arrow.
  5. Write a retry decorator with exponential backoff. — Know the three-layer pattern, functools.wraps, and exponential delay with jitter.
  6. How do you test a pipeline depending on S3, API, and DB? — moto for S3, responses for API, pytest fixtures with test DB. Mock at boundary, not internals.
  7. ETL vs ELT — when choose each? — ETL: transform before load (sensitive data, schema enforcement). ELT: load raw, transform in warehouse (dbt, SQL) — preferred for cloud DWs.

MetLife: PySpark + 20 Sources

"You ingest from 20+ sources. One sends malformed JSON. How do you prevent it from breaking the entire pipeline?"▶

Why they ask this: Tests defensive programming and fault isolation in production pipelines.

Answer framework:

  1. PERMISSIVE mode: spark.read.option('mode','PERMISSIVE').option('columnNameOfCorruptRecord','_corrupt_record') — captures bad rows instead of failing.
  2. Quarantine pattern: Route records where _corrupt_record IS NOT NULL to a quarantine Delta table. Alert on row count via Azure Monitor.
  3. Source isolation: Each of the 20 sources should be a separate pipeline/stage. One source failure doesn't cascade. Use ADF ForEach or Databricks Workflows with independent tasks.

Accenture: Kafka + Autoloader

"You built streaming pipelines from Confluent Kafka. How did you ensure exactly-once semantics end-to-end?"▶

End-to-end exactly-once requires three components:

  1. Source (Kafka): Replayable — consumer tracks offsets, can replay from any point.
  2. Processing (Spark Structured Streaming): Checkpoint directory on ADLS tracks committed offsets + processing state per micro-batch. On failure, replays from last committed offset.
  3. Sink (Delta Lake): Idempotent — Delta's transaction log prevents duplicate writes from replayed micro-batches.

Key configs:

df = spark.readStream \
    .format("kafka") \
    .option("kafka.bootstrap.servers", "confluent-broker:9092") \
    .option("subscribe", "claims-topic") \
    .option("startingOffsets", "latest") \
    .option("kafka.group.id", "claims-consumer-group") \
    .load()

df.writeStream \
    .format("delta") \
    .option("checkpointLocation", "/checkpoints/claims/") \
    .outputMode("append") \
    .trigger(processingTime="30 seconds") \
    .start("/silver/claims/")
Key Point

"Exactly-once" is really "effectively-once" — messages may be delivered multiple times, but the effect (state change) is applied exactly once. The checkpoint + replayable source + idempotent sink is what delivers this guarantee.

Key design decisions to discuss:

  • Grain: One row per claim (transaction fact). Declare grain first — drives everything else.
